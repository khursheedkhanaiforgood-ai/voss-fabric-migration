EXOS/SwitchEngine → VOSS/FabricEngine Migration

A hands-on migration of a two-switch, two-AP lab from EXOS SwitchEngine 33.x to VOSS/FabricEngine — rebuilding VLANs as I-SIDs, routing via SPB, and preserving full XIQ-managed AP3000 connectivity across Corp, Guest, Corp2, and Guest2 SSIDs. Every step references the completed EXOS deployment as the learning baseline — same hardware, same XIQ org, same Quantum Fiber network.

Starting Point — EXOS Lab (April 2026)

Deployed Lab — 2 Switches · 2 APs · 4 SSIDs

  • SW1 — 5320-16P-2MXT-2X @ 192.168.0.28 · VLANs 20 (Corp) + 30 (Guest) · AP3000 on port 3
  • SW2 — 5320-16P-2MXT-2X @ 192.168.0.11 · VLANs 50 (Corp2) + 60 (Guest2) · AP3000 on port 3
  • VLAN 10 — SW_VLAN_10_Mgmt · SVIs 10.10.0.1 / 10.10.0.2 · Inter-switch management
  • DHCP — CLI-configured on both switches · XIQ manages VLANs/port types/AP policies
  • Internet — Verified on all 4 SSIDs via Quantum Fiber + static routes
EXOS vs VOSS/FabricEngine — Key Differences
ConceptEXOS / SwitchEngineVOSS / FabricEngine
CLI modeEXOS flat CLIVSP hierarchical CLI (acli / vlan / router isis)
VLANsNamed VLANs (create vlan)VLANs + I-SIDs (Service Instance IDs for SPB fabric)
L3 routingenable ipforwarding vlan XIP interfaces under VLAN + VRF-aware routing
DHCP serverconfigure vlan X dhcp-address-rangeip dhcp-server under interface VLAN
Default routeAuto via DHCP/BOOTP (bo origin)Static or OSPFv2/BGP in VRF GlobalRouter
FabricNo fabric — pure L2/L3SPB (IEEE 802.1aq) IS-IS-based fabric
ManagementXIQ ZTP+ via VLAN 1XIQ ZTP+ via VLAN 1 (same)
Factory resetdelete /intflash/config.cfg → bootreset or delete nvram config
Port configEXOS port types via XIQPort types via XIQ (same XIQ workflow)
